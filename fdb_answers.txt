Q1: The reason that foundationDB decouples these two components is so each layer can scale and recover by itself. Since the transaction management system handles things such as read versions and conflict checking, while the storage system stores key value data and serves reads this separation between the two keeps the design modular and lets foundationDB optimize transactional logic separately from physical data storage.

Q2: The resolvers are responsible for checking read write conflicts between concurrent transactions. The conflict check happens after the transaction has been assigned a commit version but before the commit is fully accepted and if a conflict is detected the transaction is aborted and needs to retry.

Q3: The read version defines a fixed snapshot of the database that a transaction reads from. Every committed write in is assigned a version and a transaction is only allowed to read values with versions less than or equal to its read version. Because all reads within the transaction use the same version cutoff they all have a consistent view of the database from a single point in time. This prevents mixing old and new data from concurrent transactions and guarantees isolation even while other transactions are committing at the same time.

Q4: Snapshot reads differ from standard reads as they do not add entries to the transaction’s read conflict set. That allows snapshot reads to observe data without causing the transaction to conflict with concurrent writes. I would use a snapshot read over a standard read when you want to inspect data for a background analysis but do not want those reads to interfere with conflict detection.

Q5: Recovery is cheap in foundationDB because LogServers durably record committed changes during normal operation while StorageServers apply those updates constantly in the background. Since the logs already contain the durable history and storage is kept consistently up to date new components can catch up from the logs and resume service quickly.
